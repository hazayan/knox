# Secret Rulebook

This rulebook defines how Knox secrets are named, requested, and referenced by
operators and automation.

## Scope

Knox is the source of truth for secret material used by local infrastructure
automation. Kha and scripts may reference Knox secret names, but they must not
store or log secret values.

Secret examples include:

- API tokens
- TLS private keys
- repository signing keys
- bootstrap passwords
- service account credentials
- encrypted backup passphrases

## Request Workflow

When automation needs a secret that is not already present, the agent must stop
and ask the operator to create it in Knox. The agent must not ask the operator
to paste the secret into chat.

The request must include:

- the proposed Knox secret name
- what the secret is used for
- whether the secret is generated by the operator or copied from a provider
- the expected format, without revealing any real value
- a one-line command template

Template:

```sh
printf '%s' '<secret-value>' | knox key create '<secret-name>'
```

For generated values:

```sh
openssl rand -base64 32 | knox key create '<secret-name>'
```

After the operator confirms that the secret exists, automation resumes by
referencing only the secret name.

## Naming Format

Secret names use colon-separated segments:

```text
<scope>:<system>:<environment>:<purpose>[:<qualifier>]
```

Rules:

- use lowercase ASCII
- use `a-z`, `0-9`, and `-` inside each segment
- use `:` only as a segment separator
- avoid private values in names, such as passwords, tokens, IP addresses, or
  user secrets
- prefer stable service names over hostnames unless the secret is host-bound
- do not include timestamps except for intentionally retained historical keys
- rotate by creating a version of the same key when the semantic role is the
  same
- create a new key name when the trust boundary or purpose changes

## Scopes

Recommended top-level scopes:

```text
infra
service
host
user
provider
backup
build
```

Use `infra` for shared infrastructure concerns, `service` for a logical
application, `host` for host-bound material, `provider` for external provider
credentials, and `build` for packaging/release automation.

## Environments

Recommended environment segment values:

```text
prod
staging
dev
test
bootstrap
```

Use `bootstrap` for temporary or first-install secrets that should be rotated or
retired after steady-state access exists.

## Purposes

Recommended purpose segment values:

```text
api-token
client-secret
private-key
signing-key
bootstrap-password
backup-passphrase
ca-key
tls-key
pin
recovery-key
```

Prefer concrete names. Avoid generic names like `secret`, `key`, or `token`
unless they are qualified by a clear purpose.

## Examples

Cloudflare API token for ACME DNS-01:

```text
provider:cloudflare:prod:api-token:acme-dns01
```

Kanidm bootstrap credentials on the primary identity host:

```text
service:kanidm:bootstrap:bootstrap-password:admin
service:kanidm:bootstrap:bootstrap-password:idm-admin
```

TLS private key managed for Kanidm:

```text
service:kanidm:prod:tls-key
```

FreeBSD repository signing key:

```text
build:foji-bsd:prod:signing-key
```

Host-bound FIDO2 PIN:

```text
host:identity-a:prod:pin:fido2
```

Kanidm backup artifact passphrase:

```text
backup:kanidm:prod:backup-passphrase
```

## Kha References

Kha manifests should reference Knox secrets by name only:

```toml
[secret]
provider = "knox"
name = "provider:cloudflare:prod:api-token:acme-dns01"
```

Kha must treat secret values as sensitive:

- do not print values in logs
- do not store values in plans or state exports
- redact values in errors
- pass values through temporary files or process environments only when the
  target tool requires it
- clean up temporary files after use

## Rotation

When the semantic role is unchanged, rotate by adding a new version to the same
Knox key name. Examples:

```sh
printf '%s' '<new-secret-value>' | knox key rotate '<secret-name>'
```

When the trust boundary changes, create a new key name. Examples of trust
boundary changes include moving from bootstrap to production, changing provider
accounts, or splitting one shared credential into per-service credentials.

## Bootstrap Secrets

Bootstrap secrets must be treated as temporary unless explicitly documented
otherwise. Once steady-state authentication exists:

1. create the durable production secret,
2. update dependent resources,
3. verify the new path,
4. rotate or delete the bootstrap secret.

## Operator Prompt Standard

Agents should use this phrasing when a new secret is required:

```text
Please create this Knox secret, then tell me when it exists:

name: <secret-name>
purpose: <short purpose>
format: <expected format>
command:
printf '%s' '<secret-value>' | knox key create '<secret-name>'
```

For generated secrets:

```text
Please create this generated Knox secret, then tell me when it exists:

name: <secret-name>
purpose: <short purpose>
format: base64 random 32 bytes
command:
openssl rand -base64 32 | knox key create '<secret-name>'
```
